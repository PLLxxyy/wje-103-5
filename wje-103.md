# 甜品团购社区 — 全栈Web应用

## 一、硬性约束

1. **前后端分离**：前端 Vue 3 + TypeScript + Vant（移动端UI）+ Vite，后端 Node.js + Express + TypeScript + Prisma，数据库 MySQL。前后端通过 RESTful API 通信。
2. **Docker Compose 一键部署**：提供 `docker-compose.yml`（不写 `version` 字段，顶层 `name` 字段），使用 `.env` 管理所有环境变量，`docker compose up -d` 即可完整启动。
3. **COMPOSE_PROJECT_NAME**：`wjesweetgroup`
4. **端口映射**：前端 `28603:80`，后端 `29603:3000`，数据库 `57603:3306`
5. **数据库类型**：MySQL 8.0
6. **所有实体必须贯穿全栈**：从 Prisma schema → Service → Controller → Route → API 调用 → Store → 页面组件，缺一不可。
7. **横切关注点必须贯穿所有指定层**：认证授权和 RBAC 权限控制必须在每一层都有对应实现，不得遗漏。
8. **共享组件/hooks/枚举必须被多处引用**：不得出现只被一个页面使用的"共享"组件。
9. **测试样例数据必须可运行**：`prisma/seed.ts` 必须能成功执行，预置数据覆盖所有核心实体。
10. **严禁硬编码**：数据库连接串、JWT 密钥、端口号等全部通过环境变量注入。
11. **代码注释使用中文**。
12. **移动端优先**：使用 Vant 组件库，页面布局以移动端体验为主。

## 二、项目主要需求

### 2.1 项目概述

甜品团购社区是一个面向甜品爱好者的移动端Web应用。用户可以浏览入驻的甜品店铺和甜品，团长可以发起团购活动，团员通过接龙方式参与团购，还可以通过投票功能决定下一次团购的口味/品类。系统支持多提货点管理，团长可以按口味和提货点统计接龙情况并管理提货流程。

### 2.2 核心实体

共 7 个核心实体，贯穿全栈（Prisma Schema → Service → Controller → Route → 前端 API → Store → 页面）：

#### 2.2.1 User（用户）

- **字段**：`id`（主键）、`nickname`（昵称）、`phone`（手机号，唯一）、`avatar`（头像URL）、`role`（枚举：member/leader/admin）、`password_hash`（密码哈希）、`created_at`（创建时间）
- **关系**：创建多个 GroupBuy（作为 leader），参与多个 JoinRecord
- **触达层**：Prisma schema users 表 → `user.service.ts` → `user.controller.ts` → `user.ts` 路由 → `api/user.ts` → `useUserStore.ts` → 页面组件

#### 2.2.2 GroupBuy（团购）

- **字段**：`id`（主键）、`title`（团购标题）、`description`（描述）、`shop_id`（外键→Shop）、`leader_id`（外键→User）、`original_price`（原价）、`group_price`（团购价）、`min_quantity`（起购数量）、`deadline`（截止时间）、`status`（枚举：recruiting/closed/delivering/completed/cancelled）、`pickup_notes`（提货说明）、`created_at`
- **关系**：属于 User（leader），属于 Shop，拥有多个 JoinRecord，拥有多个 PickupPoint，拥有多个 VoteOption
- **触达层**：Prisma schema → `groupbuy.service.ts` → `groupbuy.controller.ts` → `groupbuy.ts` 路由 → `api/groupbuy.ts` → `useGroupBuyStore.ts` → 首页/团购详情页/团购创建页

#### 2.2.3 Shop（甜品店铺）

- **字段**：`id`（主键）、`name`（店铺名称）、`description`（描述）、`address`（地址）、`contact_phone`（联系电话）、`logo`（Logo URL）、`created_at`
- **关系**：拥有多个 GroupBuy，拥有多个 DessertItem
- **触达层**：Prisma schema → `shop.service.ts` → `shop.controller.ts` → `shop.ts` 路由 → `api/shop.ts` → `useShopStore.ts` → 店铺列表页/店铺详情

#### 2.2.4 JoinRecord（接龙记录）

- **字段**：`id`（主键）、`group_buy_id`（外键→GroupBuy）、`user_id`（外键→User）、`flavor`（口味/规格）、`quantity`（数量）、`pickup_point_id`（外键→PickupPoint）、`picked_up`（是否已提货，默认 false）、`joined_at`（参与时间）
- **关系**：属于 GroupBuy，属于 User，关联 PickupPoint
- **触达层**：Prisma schema → `join.service.ts` → `join.controller.ts` → `join.ts` 路由 → `api/join.ts` → 团购详情页/订单管理页/我的接龙页

#### 2.2.5 PickupPoint（提货点）

- **字段**：`id`（主键）、`group_buy_id`（外键→GroupBuy）、`address`（详细地址）、`available_time`（开放时间）、`max_capacity`（最大容量）
- **关系**：属于 GroupBuy，被多个 JoinRecord 引用
- **触达层**：Prisma schema → 包含在 groupbuy 相关 Service/Controller 中 → 团购详情页/接龙表单/订单管理页

#### 2.2.6 VoteOption（投票选项）

- **字段**：`id`（主键）、`group_buy_id`（外键→GroupBuy）、`option_text`（选项文本）
- **关系**：属于 GroupBuy，拥有多个 VoteRecord
- **触达层**：Prisma schema → `vote.service.ts` → `vote.ts` 路由 → `api/vote.ts` → 团购详情页投票区域

#### 2.2.7 VoteRecord（投票记录）

- **字段**：`id`（主键）、`vote_option_id`（外键→VoteOption）、`user_id`（外键→User）
- **关系**：属于 VoteOption，属于 User
- **触达层**：Prisma schema → `vote.service.ts` → `vote.ts` 路由 → `api/vote.ts` → 团购详情页投票区域

#### 2.2.8 DessertItem（甜品条目）

- **字段**：`id`（主键）、`shop_id`（外键→Shop）、`name`（甜品名称）、`description`（描述）、`price`（价格）、`image`（图片URL）、`category`（分类）
- **关系**：属于 Shop
- **触达层**：Prisma schema → 包含在 shop 相关 Service/Controller 中 → 店铺详情页

### 2.3 核心页面

共 6 个核心页面，每个页面必须消费至少 2 个核心实体：

| 页面路径 | 页面名称 | 消费实体 | 说明 |
|---|---|---|---|
| `/` | 首页 | GroupBuy + Shop + User | 展示进行中的团购列表，带店铺信息和倒计时 |
| `/groupbuy/:id` | 团购详情 | GroupBuy + JoinRecord + PickupPoint + VoteOption + VoteRecord | 查看团购详情、接龙列表、投票区域，填写接龙表单 |
| `/groupbuy/create` | 发起团购 | Shop + GroupBuy + PickupPoint | 团长填写团购信息（选择店铺、设置价格、添加提货点） |
| `/order/manage` | 订单管理 | JoinRecord + GroupBuy + PickupPoint + User | 团长查看接龙汇总，按口味/提货点统计，标记已提货 |
| `/shops` | 甜品店铺 | Shop + GroupBuy + DessertItem | 浏览入驻店铺，查看甜品列表和历史团购记录 |
| `/my/joins` | 我的接龙 | JoinRecord + GroupBuy + User | 用户查看自己参与的所有接龙记录 |

### 2.4 横切关注点

#### 2.4.1 认证授权（JWT）

触达层清单（每一层都必须有对应实现）：
- **DB**：`users` 表 `password_hash` 字段
- **后端中间件**：`middlewares/auth.ts` — JWT token 验证中间件
- **后端路由**：`routes/auth.ts` — 登录/注册/刷新 token 端点
- **后端配置**：`config/index.ts` — JWT_SECRET 环境变量
- **前端请求拦截器**：`utils/request.ts` — 自动附加 Authorization header，处理 401 跳转
- **前端路由守卫**：`router/index.ts` — 未登录重定向到登录页
- **前端 hooks**：`hooks/useAuth.ts` — 封装登录/登出/获取当前用户状态

#### 2.4.2 RBAC 权限控制

触达层清单：
- **DB**：`users` 表 `role` 字段（枚举：member/leader/admin）
- **后端中间件**：`middlewares/roleGuard.ts` — 角色校验中间件，如 `roleGuard(['leader', 'admin'])`
- **前端路由守卫**：`router/index.ts` — role 字段判断，非团长不可访问 `/groupbuy/create` 和 `/order/manage`
- **前端权限指令**：`directives/v-permission.ts` — `v-permission="['leader']"` 控制按钮显隐
- **前端类型**：`types/permissions.ts` — 权限相关类型定义

### 2.6 共享前端组件

| 组件名 | 功能 | 引用页面 |
|---|---|---|
| `<GroupBuyCard>` | 团购卡片（展示标题、价格、倒计时、店铺名、状态标签） | 首页、店铺详情页、我的接龙页 |

### 2.7 共享 Hooks

| Hook 名 | 功能 | 引用位置 |
|---|---|---|
| `useAuth` | 认证状态管理（login/logout/currentUser/token） | 路由守卫、导航栏、所有需要身份验证的页面 |
| `useCountdown` | 倒计时逻辑（接收 deadline，返回天/时/分/秒，自动刷新） | CountdownTimer 组件、团购详情页 |

### 2.8 后端中间件

| 中间件 | 文件 | 功能 |
|---|---|---|
| `authMiddleware` | `middlewares/auth.ts` | 解析 Authorization header，验证 JWT token，注入 req.user |
| `roleGuard` | `middlewares/roleGuard.ts` | 接收允许的角色数组，校验 req.user.role 是否在允许列表中 |
| `errorHandler` | `middlewares/errorHandler.ts` | 统一错误处理，捕获所有未处理异常，返回标准化错误响应 |
| `requestLogger` | `middlewares/requestLogger.ts` | 记录请求方法、路径、状态码、响应时间 |

### 2.9 技术要求

1. **前端路由**：使用 Vue Router，包含路由守卫（认证 + 角色）
2. **状态管理**：使用 Pinia
3. **HTTP 客户端**：使用 Axios，封装统一请求工具（`utils/request.ts`）
4. **ORM**：使用 Prisma，schema 定义所有实体和关系
5. **API 规范**：RESTful 风格，统一响应格式 `{ code, data, message }`
6. **错误处理**：后端统一错误处理中间件，前端统一错误提示（使用 Vant Toast）
7. **日期处理**：使用 dayjs 处理截止时间和倒计时
8. **密码加密**：使用 bcrypt
9. **移动端适配**：Vant 组件 + rem/vw 适配方案

### 严禁合并职责到单一文件

- 每个 Service 文件只处理一个实体的业务逻辑
- 每个 Controller 文件只处理一个实体的请求/响应
- 每个 Route 文件只定义一个实体的路由
- 每个 Store 文件只管理一个实体的状态
- 每个 API 文件只封装一个实体的接口调用
- 每个 Type 文件只定义一个实体的类型
- 中间件不得与业务逻辑混写
- 组件必须是纯粹的展示/交互组件，不得包含业务逻辑

## 四、核心流程

### 4.1 发起团购 → 分享 → 接龙 → 截止 → 统计 → 提货

```
1. 团长登录 → 进入"发起团购"页面
   ↓
2. 选择甜品店铺 → 填写团购信息（标题、描述、原价、团购价、起购数量、截止时间）
   ↓
3. 添加提货点（地址、开放时间、最大容量）
   ↓
4. 可选：添加投票选项（口味/品类），发起投票让团员选择
   ↓
5. 提交团购 → 状态为 recruiting → 分享给团员
   ↓
6. 团员打开团购详情 → 查看团购信息和接龙列表
   ↓
7. 团员参与接龙（选择口味/规格、数量、提货点）
   ↓
8. 可选：团员参与投票
   ↓
9. 到达截止时间 → 系统/团长将状态变更为 closed
   ↓
10. 团长在订单管理页面查看接龙汇总
    - 按口味统计：每种口味的总数量
    - 按提货点统计：每个提货点的订单数
    ↓
11. 状态变更为 delivering → 团长配送/备货
    ↓
12. 团员到提货点提货 → 团长标记已提货
    ↓
13. 所有订单提货完成 → 状态变更为 completed
```

### 4.2 投票流程

```
1. 团长创建团购时添加投票选项（如：草莓蛋糕、芒果千层、抹茶慕斯）
   ↓
2. 投票区域展示在团购详情页
   ↓
3. 团员点击投票（每人每组投票限一票）
   ↓
4. 实时显示各选项得票数和比例
   ↓
5. 团长根据投票结果决定正式团购的甜品
```

## 六、代码存放规则

1. **前端代码**：全部存放在 `frontend/` 目录下
2. **后端代码**：全部存放在 `backend/` 目录下
3. **根目录文件**：`docker-compose.yml`、`.env.example`、`.gitignore`、`README.md`
4. **前后端不得共享代码目录**：类型定义、枚举等需要分别在前后端各维护一份
5. **Prisma 相关**：`backend/prisma/schema.prisma` 和 `backend/prisma/seed.ts`
6. **环境变量模板**：根目录 `.env.example` 列出所有必需变量及示例值
7. **Dockerfile**：`frontend/Dockerfile` 和 `backend/Dockerfile` 各自独立

## 七、部署要求

### 7.1 Docker Compose 模板（MySQL 版）

```yaml
name: wjesweetgroup

services:
  db:
    image: mysql:8.0
    container_name: ${COMPOSE_PROJECT_NAME:-wjesweetgroup}-db
    restart: unless-stopped
    environment:
      MYSQL_ROOT_PASSWORD: ${DB_ROOT_PASSWORD}
      MYSQL_DATABASE: ${DB_NAME}
      MYSQL_USER: ${DB_USER}
      MYSQL_PASSWORD: ${DB_PASSWORD}
    ports:
      - "${DB_PORT:-57603}:3306"
    volumes:
      - db_data:/var/lib/mysql
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost", "-u", "root", "-p${DB_ROOT_PASSWORD}"]
      interval: 5s
      timeout: 5s
      retries: 20

  backend:
    build: ./backend
    container_name: ${COMPOSE_PROJECT_NAME:-wjesweetgroup}-backend
    restart: unless-stopped
    environment:
      DATABASE_URL: mysql://${DB_USER}:${DB_PASSWORD}@db:3306/${DB_NAME}
      JWT_SECRET: ${JWT_SECRET}
    ports:
      - "${BACKEND_PORT:-29603}:3000"
    depends_on:
      db:
        condition: service_healthy

  frontend:
    build: ./frontend
    container_name: ${COMPOSE_PROJECT_NAME:-wjesweetgroup}-frontend
    restart: unless-stopped
    ports:
      - "${FRONTEND_PORT:-28603}:80"
    depends_on:
      - backend

volumes:
  db_data:
```

### 7.2 .env.example

### 7.3 后端 Dockerfile 要求

- 基于 `node:18-alpine`
- 复制 `package.json` 和 `package-lock.json`，执行 `npm install`
- 复制全部源码
- 执行 `npx prisma generate`
- 启动命令：先执行 `npx prisma db push`（确保表结构同步）和 `npx prisma db seed`（填充测试数据），再启动服务
- 暴露 3000 端口

### 7.4 前端 Dockerfile 要求

- 构建阶段：基于 `node:18-alpine`，执行 `npm install` 和 `npm run build`
- 运行阶段：基于 `nginx:alpine`，复制构建产物到 `/usr/share/nginx/html`
- 复制自定义 `nginx.conf`（处理 Vue Router history 模式）
- 暴露 80 端口

### 7.5 Prisma schema 注意事项

- 使用 `mysql` 作为 provider
- 所有枚举使用 Prisma 的 `enum` 语法
- 关系字段使用 `@relation` 注解
- 使用 `@@map` 将模型名映射为下划线风格的表名（如 `@@map("group_buys")`）
- 使用 `@default(now())` 设置时间戳默认值
- 使用 `@default(false)` 设置 boolean 默认值
- 使用 `@default(uuid())` 或 `@default(cuid())` 生成主键
